# Дорожная карта Gamma Agent Runtime

---

## 🌌 Видение

**Gamma Agent Runtime** — это специализированная платформа исполнения агентов (Agent Runtime Platform): событийно-ориентированная, stateful-среда для оркестрации долгоживущих LLM-агентов. Платформа решает две фундаментальные проблемы, которые делают современные ИИ-системы ненадёжными в производственной эксплуатации: **непрозрачность агентного поведения** («чёрный ящик») и **отсутствие персистентного состояния** (амнезия между запросами).

В отличие от традиционного подхода «запрос → ответ → забыть», Gamma Agent Runtime трактует каждый агент как **долгоживущий daemon-процесс** с управляемым жизненным циклом, собственным пространством памяти и детерминированным контекстом. Агенты не изолированы — они коммуницируют через **шину событий (Event Bus) на базе Redis Streams**, которая служит единым источником истины для всей агентной экосистемы. Платформа является предметом **магистерской диссертации (MSc)** в области Human-Computer Interaction и архитектуры AI-систем.

---

## 🧠 Два фундаментальных столпа платформы

### Столп I — Agent Observability Layer: от «чёрного ящика» к «белому ящику»

Современные LLM-агенты — это в полном смысле **«чёрные ящики»**: разработчик не видит, какой системный промпт был собран в момент инициализации, какие инструменты (Tools / Syscalls) были вызваны, сколько токенов контекстного окна израсходовано и почему агент принял то или иное решение. Это фундаментальный барьер для надёжности, отладки и научного анализа агентных систем.

**Observability Layer** платформы вводит полную трассировку агентного поведения в реальном времени:

- **Context Transparency** — полная видимость того, *какой именно* `system_prompt` получил агент при инициализации сессии: структура блоков контекста (глобальный промпт рантайма, `context.md` приложения, `agent-prompt.md` персоны, описание инструментов), происхождение каждого блока и оценка занятости контекстного окна.
- **Syscall Tracing** — каждый вызов инструмента (Tool Use / Syscall) логируется как структурированное событие с входными параметрами, результатом и временными метками; история вызовов доступна для реплея и анализа.
- **Real-time Token Accounting** — инкрементальный учёт `input_tokens`, `output_tokens`, `cache_read_tokens` и `cache_write_tokens` на уровне каждого запуска агента и агрегированно по сессии; база для оптимизации промптов и обоснования архитектурных решений.
- **Session Lifecycle Monitor** — визуальный диспетчер всех активных агентных сессий с атрибутами статуса (`Idle / Running / Thinking / Error / Aborted`), типа (`Transient / Daemon`) и управляющими действиями (Kill, Inspect).

### Столп II — Long-lived Agent Runtime Environment: stateful экосистема

Стандартная модель взаимодействия с LLM — **stateless request-response**: каждый вызов изолирован, контекст передаётся заново, память не сохраняется между сессиями. Эта модель принципиально неприменима для автономных агентов, которые должны работать фоново, накапливать знания и координировать действия.

**Agent Runtime Environment** платформы реализует полноценную stateful-экосистему:

- **Агенты как daemon-процессы:** каждый агент существует как долгоживущий процесс с управляемым жизненным циклом. Закрытие UI-окна не завершает агент — оно лишь отключает визуальное представление. Агент продолжает работу в daemon-режиме, сохраняя сессию, контекст и историю.
- **Event Bus (Memory Bus) на Redis Streams:** все события агентной экосистемы — запуски, завершения, вызовы инструментов, межагентные сообщения — публикуются в `gamma:memory:bus`. Шина обеспечивает event-sourcing, детерминированный реплей, кросс-агентный дебаггинг и аудит всей истории взаимодействий.
- **Персистентная память приложений:** каждое приложение располагает изолированным key-value хранилищем поверх Redis (`useAppStorage`); состояние переживает перезагрузки, обновления бандлов и рестарты рантайма.
- **Фоновое выполнение задач:** агенты могут регистрировать задачи с Cron-расписанием или событийными триггерами, выполняемые независимо от активности пользователя — с наследованием разрешений из контекста создания задачи.

## 🔑 Ключевые концепции

- **LLM как сопроцессор:** LLM используется как программируемый сопроцессор для планирования, декомпозиции задач и высокоуровневого рассуждения — не как монолитный диалоговый интерфейс.
- **Context Injection:** механизм детерминированной сборки и инъекции контекста (`system_prompt`, `context.md`, `agent-prompt.md`) при инициализации сессии — ключевой примитив Observability Layer и управления поведением агента.
- **ACA (Agent Capability Architecture):** формальная архитектура способностей агентов: Capabilities как модульные, инъецируемые единицы поведения вместо монолитных промптов; Syscalls как типизированный контракт взаимодействия агента с ядром рантайма.
- **Окно = viewport в процесс:** UI-окно — это представление сессии конкретного агентного процесса. Его жизненный цикл не совпадает с жизненным циклом агента, что обеспечивает истинный daemon-режим и бесшовное переподключение.

## ✅ Phase 1 — Бэкенд-микроядро

- **Архитектура NestJS:** Модульный бэкенд на NestJS играет роль микроядра, предоставляя чётко определённый API для управления жизненным циклом приложений, компиляции бандлов и оркестрации агентов.
- **Управление состоянием в Redis:** Временное и полупостоянное состояние ОС (сессии, экземпляры приложений, записи шины памяти и т.п.) хранится в Redis, что обеспечивает малую задержку, простое горизонтальное масштабирование и чёткое разделение контуров.
- **Шина памяти на Redis Streams (Memory Bus):** Redis Streams используются как основная системная шина событий и сообщений между агентами и сервисами ОС, поддерживая event‑sourcing, реплей истории, кросс‑агентный дебаггинг и наблюдаемость архитектуры на уровне целой ОС.
- **Scaffold Service и генерация бандлов:** Сервис Scaffold может синтезировать полные бандлы приложений (React‑компоненты `.tsx` + `context.md` + `agent-prompt.md`) из высокоуровневых спецификаций, давая агентам первоклассный механизм для создания, доработки и владения микро-приложениями.

## ✅ Phase 2 — Мост OpenClaw Gateway

- **Безопасный SSE‑стриминг:** Укреплённый Gateway-компонент соединяет ОС с OpenClaw, предоставляя серверные события (SSE) для токенового стриминга от LLM с жёсткой изоляцией сессий.
- **Кумулятивный рендеринг текста:** Протокол стриминга поддерживает кумулятивные обновления текста, обеспечивая отзывчивый UX для бесед с агентами и живой генерации приложений.
- **Надёжное управление сессиями:** Сессии Gateway жёстко связаны с сессиями ОС, с чёткими границами вокруг аутентификации, контроля частоты запросов и использования ресурсов, что позволяет безопасно мультиплексировать множество параллельных взаимодействий с агентами.

## ✅ Phase 3 — OS Foundation & App Bundles

- **Настольная среда (Desktop):** Браузерный «рабочий стол» отображает окна, уведомления и служит основным уровнем взаимодействия пользователя с приложениями, которыми владеют агенты.
- **Минималистичная тема Blue Light:** Единая тема ОС (цвета, типографика, отступы, фокус‑состояния) задаёт профессиональный, читаемый базовый стиль для всех приложений, удерживая визуальное поле спокойным даже при длительных сессиях работы агентов.
- **Верхнее меню (Menu Bar):** Постоянная панель меню предоставляет глобальные элементы управления (состояние сессии, активные агенты/приложения, системные действия) и выступает как «UI‑оболочка» ядра для пользователя.
- **Динамический оконный менеджер с `DynamicAppRenderer`:** Гибкий оконный менеджер интегрирует `DynamicAppRenderer`, который динамически подгружает и монтирует бандлы приложений в окна, поддерживая создание, уничтожение и перестановку приложений без полной перезагрузки страницы.
- **Иерархия агентов:** Глобальный привилегированный агент (System Architect) обладает системным взглядом на ОС и оркестрирует многоприложечные рабочие процессы. Каждое приложение имеет изолированного «владельца»-агента с ограниченным контекстом (собственный бандл, хранилище и права).
- **API хранилища приложений (`useAppStorage`):** Первоклассный React‑хук абстрагирует постоянное хранилище данных приложения поверх Redis. Приложения используют `useAppStorage` как локальное key‑value или документное хранилище для состояния (настройки, история, черновики, кэш вычислений), пережившего перезагрузки и инкрементные обновления бандла.

> **Статус:** Конвейер генерации и монтирования бандлов, оконный менеджер и базовая интеграция с OpenClaw работают в продакшен‑режиме. Context Injection для `app-owner-{appId}` перенесён в Phase 4 и описан в `docs/backlog/app-owner-context-injection.md`.

## 🚀 Phase 4 — Agent Observability & Control (White Box Layer)

Phase 4 вводит выделенный **Agent Control Plane** для Gamma OS — слой для аудита, трассировки, анализа, создания и управления ИИ‑агентами по всей системе. Цель — сделать активность агентов *наблюдаемой, управляемой и воспроизводимой*, а не «чёрным ящиком».

Подробный план реализации: [`docs/plans/stage-4.md`](./plans/stage-4.md)

- **UI Task Manager (Agent Monitor)**
  - Визуальный диспетчер, показывающий активные и недавние сессии (например, `main`, `app-owner-notes`, `app-owner-browser`).
  - Атрибуты на сессию: статус (**Idle / Running / Thinking / Error / Aborted**), тип (**Transient** vs **Daemon**), `sessionKey`, `windowId`, `appId`, `agentId`, время создания и последней активности.
  - Управляющие действия: ручной **Kill** (через `sessions.abort` / `sessions.delete`), быстрый переход к инспекции контекста для выбранной сессии.

- **Context Debugger**
  - Инструмент разработчика для просмотра того, *какой именно* `system_prompt` и контекст получает агент в реальном времени.
  - Подсветка происхождения блоков контекста: глобальный промпт ОС, app‑owner контекст (`context.md`, `agent-prompt.md`), описание инструментов и системных ограничений.
  - Диагностика контекст‑окна: оценка использования токенов контекстом относительно максимума модели, флаги при попытке инъекции слишком больших `*App.tsx` или нескольких документов подряд.

- **Token Tracking & Billing**
  - Сбор телеметрии LLM‑вызовов: `prompt_tokens`, `completion_tokens`, (опционально) cache‑hits per run и per session.
  - Сохранение метрик в Redis (через `gamma:memory:bus` или отдельные метрик‑стримы) с возможностью агрегации по приложению (`appId`), типу сессии и интервалам времени.
  - База для дашбордов и алёртов: топовые потребители токенов, регрессии по расходу после релизов, обоснование оптимизаций промптов и контекста.

- **Lifecycle Management & Garbage Collection**
  - Чёткий контракт на уничтожение сессий: при закрытии окна или удалении приложения автоматически вызывать `sessions.delete` в Gateway и чистить связанные ключи (`gamma:sse:*`, `gamma:state:*`, `gamma:memory:bus`).
  - Политики простаивания: конфигурируемые idle‑таймауты для транзиентных сессий с мягким и жёстким завершением.
  - Наблюдаемость GC: логирование и метрики по созданным, удалённым и автоматически собранным «зомби‑» сессиям, интеграция с Agent Monitor.

- **Context Injection Refactor**
  - Цель: для `app-owner-{appId}` сессий надёжно инъецировать app‑специфический контекст (`context.md`, `agent-prompt.md`, основной `*App.tsx`) при инициализации сессии через единый вызов (например, `sessions.create` в Gateway) с богатым `system_prompt`.
  - Требуется единый, канонический путь инициализации сессии в OpenClaw, согласованный с текущим протоколом `gateway-ws.service.ts`.
  - Важно учитывать лимиты контекст‑окна: сырая инъекция всего `*App.tsx` и `context.md` может превысить доступный контекст при росте приложения; Phase 4 должен предусмотреть базовую минимизацию кода, токен‑каунтинг и, при необходимости, семантическое сжатие перед отправкой промпта в OpenClaw.

## Стратегические эпики: Phases 5–11

Фундамент платформы заложен в Phases 1–4. Phases 5–11 — это стратегические эпики, которые превращают Gamma OS из стабильной AI-платформы в полноценную автономную операционную систему.

---

## 🔮 Phase 5 — Слой данных и персистенция (Data Layer & Persistence)

**Видение:** Предоставить каждому приложению собственную изолированную БД SQLite и единую абстракцию доступа к данным через **Виртуальную файловую систему (VFS)**. Агенты взаимодействуют с данными через единый интерфейс независимо от бэкенда — SQLite, Redis или локальное хранилище; изоляция и консистентность гарантируются на уровне VFS.

**Техническая реализация:**
1. Встроенные экземпляры SQLite на уровне приложения с изоляцией по неймспейсу.
2. VFS как унифицированный слой абстракции над SQLite, Redis и локальным storage — единые операции read/write/query.
3. Drag‑and‑drop файлов (CSV, логи, телеметрия) в окно приложения с автоматическим выводом схемы и созданием таблиц.

**Ценность:** Высокопроизводительная среда для дата‑ориентированных инструментов; переключаемость бэкендов без изменения кода агента; загрузка, анализ и визуализация внутри изолированного контекста приложения.

---

## 🔮 Phase 6 — Архитектура способностей агентов (ACA — Agent Capability Architecture)

**Видение:** Решить проблему «взрыва сложности промптов»: личность агента и его операционные возможности разделены; тяжёлая логика выносится из монолитного `agent-prompt.md` в модульные способности (Capabilities). Определение того, *как* агенты используют драйверы и системные вызовы, предшествует их защите и межагентному взаимодействию.

**Техническая реализация:**
1. Реестр системных вызовов (Syscalls) как формальный слой взаимодействия с ядром.
2. Инъекция только необходимых инструментов (Tools) и способностей в зависимости от контекста.
3. Типизированные интерфейсы и адаптеры на уровне ядра вместо логики в промпте.

**Ценность:** Повышение стабильности LLM‑агентов, снижение токенов, обучение новым навыкам через доопределение способностей без переписывания «личности».

---

## 🔮 Phase 7 — Безопасность и менеджер разрешений (Security & Permission Manager)

**Видение:** Внедрение многоуровневой системы разрешений (MAC — Mandatory Access Control) для изоляции агентов. Разрешения выдаются **на уровне способностей (Capabilities)**: конкретному агенту разрешено использовать конкретную capability в ограниченном контексте — например, «Агент X может использовать capability `Network-Fetch` в течение 5 минут».

**Техническая реализация:**
1. Иерархия доступа (Architect / App‑Owner / Sub‑Agent) с привязкой прав к Capabilities из реестра ACA.
2. **Механизм повышения привилегий (Privilege Escalation):** при запросе capability вне роли — System Notification; доступ только после явного User Approval; Time-bound или Task-bound ограничения.

**Ценность:** Безопасная «песочница»; предотвращение утечек; гранулярный контроль над использованием способностей; аудируемый процесс одобрения для чувствительных операций.

---

## 🗺️ Phase 8 — Межагентная коммуникация (IPC — Inter-Agent Communication)

**Видение:** Переход от модели «Пользователь → Агент» к модели «Агент ↔ Агент» — агенты делегируют задачи друг другу без обязательного участия пользователя. Для оркестрации требуется возможность находить агентов по их возможностям.

**Техническая реализация:**
1. Формальный протокол сообщений (From, To, Type, Task, Payload).
2. Транспорт на базе Redis Streams (канал `gamma:ipc:requests`).
3. Реестр агентов (Agent Registry) с описанием статусов и адресации.
4. **Service Discovery:** агенты могут запрашивать систему по capabilities (например, «найти агента с capability `crypto-analysis`») — запрос возвращает список подходящих агентов и их дескрипторы.

**Ценность:** Сложные цепочки делегирования, многоагентные пайплайны, оркестрация специализированных агентов; динамическое обнаружение сервисов по capabilities без жёсткой адресации.

---

## 🗺️ Phase 9 — Экосистема и распространение приложений (App Ecosystem & Distribution)

**Видение:** Стандартизировать бандлы приложений в переносимый, версионируемый формат и обеспечить Магазин приложений на рабочем столе ОС для поиска, установки и шаринга бандлов.

**Техническая реализация:**
1. Формат бандлов (`.tsx` + `context.md` + `agent-prompt.md` + метаданные) с упаковкой в архив.
2. Магазин приложений: семантический поиск, проверка совместимости, обновления.
3. Публикация, форки и клонирование бандлов агентами и пользователями.

**Ценность:** Сообщество инструментов; шеринг дашбордов и ИИ‑личностей; единая точка распространения и обновления приложений.

---

## 🕰️ Phase 10 — Автономное планирование и фоновые задачи (Autonomous Scheduling & Background Tasks)

**Видение:** Gamma OS работает 24/7: агенты выполняют задачи в фоне независимо от активности пользователя; фоновые задачи работают с правами, выданными во время создания таски.

**Техническая реализация:**
1. Реестр задач (Task Registry) с поддержкой Cron, событийных триггеров (Event-based) и условных запусков.
2. Режим «Агент-демон»: процесс продолжается в фоне при закрытом окне или браузере.
3. Очередь выполнения (Execution Queue) в Redis; наследование разрешений из контекста создания задачи.

**Ценность:** Проактивная работа — ежечасный анализ, мониторинг рынка, обновление дашбордов и уведомления без постоянного участия пользователя.

## 🕰️ Phase 11 — Самоэволюционирующее промпт-ядро (Self-Evolving Prompt Kernel)

**Видение:** Промпты — системное ядро, эволюционирующее на основе накопленных данных о поведении, ошибках и эффективности.

**Техническая реализация:**
1. Версионирование промптов (Prompt Registry) с историей и rollback.
2. Телеметрия: ошибки, расход токенов, процент успешных задач.
3. Prompt Optimizer и Patch System — автоматическая генерация патчей инструкций на основе анализа прошлых ошибок.

**Ценность:** Самообучающаяся система, становящаяся эффективнее с каждым запуском без ручного рефакторинга промптов.
