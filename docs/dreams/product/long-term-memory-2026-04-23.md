## Вводный контекст

### Три ключевых файла системы

#### 1. `apps/core/src/orchestrator/orchestrator.service.ts` (708 строк)
Это сердце всей платформы. `OrchestratorService` управляет полным жизненным циклом агентных пайплайнов: запускает лидер-агента при получении пользовательского сообщения, порождает воркеров при назначении задач (через реакцию на события `task.assigned`), отслеживает завершение всех задач раунда и автоматически пробуждает лидера со структурированным системным сообщением. Ключевые механизмы: асинхронный генератор для потоковой обработки NDJSON-чанков Claude CLI, семафорный пул процессов (`SessionPoolService`), механизм аварийной остановки (SIGTERM/SIGKILL) и детекция `taskUpdatedByAgent` через парсинг bash-команд агента. Особенно важна логика «авто-wake» лидера: когда все задачи завершены, формируется строгий протокол — система требует от лидера сначала прочитать реальные результаты через `get-task/{id}`, запрещая ответ на основе лишь заголовков задач.

#### 2. `apps/core/src/internal/internal.service.ts` (664 строки)
Нервная система агентной координации. Реализует REST API, вызываемый агентами через `curl`: назначение задач (`assign-task`), обновление статуса (`update-task`), отправка/чтение сообщений (`send-message`, `read-messages`, `broadcast`), эскалация блокеров (`report-status`), запрос ревью (`request-review`), чтение контекста проекта (`read-context`). Именно здесь живёт конечный автомат состояний задачи (`backlog → planning → in_progress → review → done/failed`), маршрутизация сообщений по имени или ID агента и автоматическое уведомление лидера при наличии блокеров. Этот модуль определяет, что агенты могут «делать» в системе в отличие от того, что они «думают».

#### 3. `apps/core/src/agents/claude-md.generator.ts` (426 строк)
Слой промпт-инжиниринга платформы. `ClaudeMdGenerator.generate()` создаёт персонализированный `CLAUDE.md` файл для каждого агента — это буквально мозговой имплантат агента: идентичность, роль, протоколы делегирования, запреты, полная документация инструментов, протокол пробуждения. Для лидера генерирует: критические правила делегирования («ты координатор, не исполнитель»), запрет busy-wait циклов, протокол Wake-Up (читай `task.result` прежде чем отвечать), шаблон декомпозиции задач в JSON. Для воркеров: политику «Team App OFF LIMITS», обязательный формат `update-task` с реальными данными. Генератор фактически кодирует организационную культуру команды агентов.

### Два внешних концептуальных стимула

#### Концепт A: Стигмергия в мультиагентных системах
*(Russell, S., Norvig, P. «Artificial Intelligence: A Modern Approach» + Bonabeau et al. «Swarm Intelligence»)*

Стигмергия — механизм непрямой координации через среду, впервые описанный у социальных насекомых. Муравьи не общаются напрямую: каждый оставляет феромонные следы в окружающей среде, на которые реагируют другие. Ключевой тезис применительно к ИИ-системам: **сложное коллективное поведение возникает не из прямой коммуникации агентов, а из модификации разделяемого состояния среды**. Агент А пишет открытие в shared-хранилище → агент B, который никогда не общался с A напрямую, считывает его и строит на нём следующий шаг. Применительно к Gamma Runtime: агенты уже имеют общий `shared/` каталог, но они не «видят», что туда писали другие агенты в прошлых сессиях. Стигмергическая система требует **персистентных, индексированных, контекстуально извлекаемых следов** — а не просто папки с файлами.

#### Концепт B: Когнитивная нагрузка и экстернализация состояния в HCI
*(Sweller, J. «Cognitive Load Theory» + Hutchins, E. «Cognition in the Wild»)*

Теория когнитивной нагрузки утверждает, что рабочая память человека жёстко ограничена (~7±2 единиц одновременно). Когда система требует от пользователя помнить состояние процессов, историю решений, прогресс агентов — когнитивная нагрузка растёт нелинейно с числом агентов. Ключевой тезис из «Cognition in the Wild»: **оптимальные когнитивные системы экстернализируют состояние в среду** — пилот самолёта смотрит на приборы, не держит всё в голове. Применительно к Gamma Runtime: пользователь сейчас вынужден «помнить», что агенты нашли в прошлых сессиях, какие решения были приняты, почему та или иная архитектура была выбрана. Если платформа сама хранит и визуализирует эволюцию коллективного знания команды — когнитивная нагрузка пользователя радикально снижается, а ценность каждого запуска умножается.

## Выжимка диалектики (10 шагов)

Десятишаговая диалектика выявила ключевую эволюцию идеи через следующие стадии:

**Шаги 1–3 (Распределённое знание):**
Гипотеза о семантических эмбеддингах как общей памяти (шаг 1) была атакована через требование векторной БД и скрытых затрат. Рефакторинг в сторону структурированного графа знаний с query-API. Идея иерархической топологии команд (шаг 3) выявила конфликт с `MAX_CONCURRENT_AGENTS` и архитектурой `runningPipelines` — упрощена до предконфигурированных иерархических шаблонов.

**Шаги 4–6 (Реплей и маршрутизация):**
Temporal Replay (шаг 4) столкнулся с фундаментальным ограничением: состояние Claude-сессии нельзя форкнуть, сессионный ID линеен. Выжила идея **checkpoint-retry** — перезапуск с отредактированным сообщением. Peer Review Protocol (шаг 5) отклонён из-за удвоения времени выполнения; выжил как «самокритика» в форматах task.result. Capability Negotiation (шаг 6) упрощён до статической валидации skill-to-task match.

**Шаги 7–10 (Проактивность и аналитика):**
Ambient Intelligence (шаг 7) разрушен угрозой feedback loops и конкуренцией за пул; выжил как «post-round health snapshot». Предобработка намерений (шаг 8) оказалась избыточной для общего случая — выжила как параметрические шаблоны задач. Context Window Segmentation (шаг 9) идентифицировал реальную проблему (агенты упираются в лимит контекста), предложил checkpoint injection через обновление CLAUDE.md. Agent Performance Analytics (шаг 10) выявил ценность паттернов провалов для лидера.

**Ключевая конвергенция:** Идеи шагов 1, 4, 7, 9 и 10 сходятся к единому принципу — **персистентное коллективное знание, пережившее границы сессии**. Именно это лежит в основе итоговой фичи.

>
> Team Long-Term Memory меняет природу продукта. Это больше не запускатель агентов — это **накопитель коллективного интеллекта**. С каждым запуском команда умнеет. Второй раунд job-hunting сразу знает, что Djinni даёт RSS, DOU требует FlareSolverr, LinkedIn бесполезен. Третий раунд знает, какие компании уже отклонили. N-й раунд работает с накопленной базой знаний в сотни записей.
>
> Техническая реализация минимальна: таблица `knowledge_entries` в Postgres, одна функция извлечения поверх существующих `task.result`, один новый endpoint в Internal API. Агентам ничего новое учить не надо — экстракция происходит автоматически. UI — просто визуализация уже собираемых данных. Это незамедлительная value delivery без архитектурной хирургии.
>
> Более того: это первая реализация стигмергической координации между **сессиями** в любом публично известном локальном multi-agent runtime. Gamma Runtime становится reference implementation.»

### Пессимист (Persona B): Атака на «Живую Память»

> «Красивая метафора. Теперь давай разберём детали.
>
> Первое: что именно извлекать? `task.result` — это JSON с полями `summary`, `filesChanged`, `autoCompleted`. Summary — неструктурированный текст от 10 до 500 символов. Ты хочешь автоматически извлечь из "Scraped 47 vacancies from Djinni, saved to project/vacancies-djinni.json" факт "Djinni доступен через RSS без FlareSolverr"? Это требует ещё одного LLM-вызова для извлечения. Ещё один LLM-вызов — это задержка, стоимость, и новая точка отказа. А без него — ты просто копируешь summary в отдельную таблицу и называешь это "базой знаний".
>
> Второе: релевантность при query. "Дай мне знания по тегу 'djinni'" работает только если агент правильно тегировал запись при создании. Но агент не создаёт записи — создаёт автоматика. Значит, теги должны извлекаться автоматически. Значит, снова LLM-вызов или regex-эвристика, которая будет ошибаться.
>
> Третье: CLAUDE.md уже 400+ строк. Секция "Prior Knowledge" с 5 записями добавит ещё ~200 строк. Ты буквально увеличиваешь входной контекст агента в каждой сессии. Проблему context window ты не решаешь — ты её усугубляешь.
>
> Четвёртое: устаревание знаний. Через 3 месяца знание "DOU доступен через FlareSolverr на порту 8191" устареет — DOU изменит структуру. Но запись в базе по-прежнему будет появляться в запросах. Нет механизма TTL или инвалидации. Стагнирующая база знаний хуже, чем её отсутствие, — она активно вводит агентов в заблуждение.
>
> Пятое: что делать со знаниями разных "эпох" проекта? Для job-hunting команды знание из апреля о вакансиях нерелевантно в октябре. Но система не знает, что это "одна и та же команда, но другой раунд поиска".»

### Арбитраж Главного Архитектора

Пессимист выиграл по пунктам 1 и 3 (LLM-вызов для извлечения и раздувание CLAUDE.md). Оптимист выиграл по пунктам о накопительной ценности и минимальности инфраструктуры.

**Ключевые синтетические решения:**
- Извлечение — **не LLM**, а детерминированная структурная экстракция: если агент использует протокол `update-task` с `summary`, то summary уже является кандидатом в запись. Дополнительно — regex для числовых артефактов (цифры, URL, пути к файлам).
- Вместо инъекции в CLAUDE.md — **отдельный endpoint** `query-knowledge`, который агент вызывает сам при необходимости. Пусть агент решает, когда ему нужна история.
- **Версионирование через `sessionId`** — каждая запись помечена сессией. UI позволяет фильтровать по «текущий проект» vs «вся история».
- **TTL и confidence degradation** — записи имеют поле `confidence` и таймштамп. Старые записи автоматически снижают confidence через 30 дней. Агент видит: `[LOW CONFIDENCE, 45d ago]`.

## Спецификация новой фичи

### Название: **Gamma Team Long-Term Memory (GTLM)**
### Подзаголовок: *Персистентная коллективная база знаний, накапливаемая командой агентов между сессиями*

---

### Обоснование

Текущий Gamma Runtime — это инструмент **«безамнезийной» оркестрации**: каждый запуск начинается с чистого листа. Agentы не помнят результатов предыдущих сессий, не знают, какие источники данных уже были исследованы, какие ошибки уже совершались, какие артефакты уже созданы. Это порождает три системных проблемы:
1. **Дублирование работы**: агенты повторно исследуют то, что уже известно.
2. **Накопление ошибок**: агенты снова наступают на те же грабли.
3. **Когнитивная нагрузка пользователя**: пользователь сам вынужден хранить историческую память и вручную инжектировать её в каждый новый запрос.

GTLM решает это через **автоматически поддерживаемую, структурированную, версионированную базу знаний** уровня команды, которая персистирует между сессиями и доступна агентам через Internal API.

### Архитектура

#### 1. Схема данных: таблица `knowledge_entries`

```sql
CREATE TABLE knowledge_entries (
    id           TEXT PRIMARY KEY,                    -- ULID с префиксом kn_
    team_id      TEXT NOT NULL REFERENCES teams(id),
    agent_id     TEXT NOT NULL REFERENCES agents(id), -- агент-автор
    task_id      TEXT REFERENCES tasks(id),           -- задача-источник (опц.)
    session_id   TEXT,                                -- Claude session ID
    kind         TEXT NOT NULL,                       -- 'finding' | 'decision' | 'artifact' | 'failure'
    content      TEXT NOT NULL,                       -- содержание записи (до 1000 символов)
    tags         TEXT[] DEFAULT '{}',                 -- теги для фильтрации
    confidence   TEXT DEFAULT 'high',                 -- 'high' | 'medium' | 'low'
    created_at   BIGINT NOT NULL DEFAULT extract(epoch from now()) * 1000,
    expires_at   BIGINT                               -- NULL = вечная запись; иначе — TTL
);
CREATE INDEX idx_knowledge_team ON knowledge_entries(team_id);
CREATE INDEX idx_knowledge_tags ON knowledge_entries USING GIN(tags);
```

#### 2. Пайплайн автоматической экстракции

После каждого события `task.completed` в `OrchestratorService` запускается детерминированный экстрактор `KnowledgeExtractorService.extractFromTaskResult()`:

```typescript
interface KnowledgeEntry {
  kind: 'finding' | 'decision' | 'artifact' | 'failure';
  content: string;
  tags: string[];
  confidence: 'high' | 'medium' | 'low';
}

// Правила экстракции (без LLM-вызовов):
// 1. Если summary содержит числовые артефакты (N записей, N файлов) → kind='artifact'
// 2. Если task.stage === 'failed' → kind='failure', confidence='high'
// 3. Если summary содержит URL/доменные имена → kind='finding', теги=[домен]
// 4. Если filesChanged.length > 0 → kind='artifact', теги=['file:'+path]
// 5. Все записи получают тег из task.kind (backend, frontend, qa и т.д.)
```

Экстрактор работает синхронно, без дополнительных LLM-вызовов. Результат — 0–5 структурированных записей на задачу.

#### 3. Internal API: новый endpoint `query-knowledge`

```
GET /api/internal/query-knowledge
  ?teamId=team_xxx
  &tags=djinni,scraping       (опционально, OR-логика)
  &kind=finding               (опционально)
  &since=1700000000000        (опционально, Unix ms)
  &limit=10                   (default: 5, max: 20)
  &confidence=high,medium     (опционально)
```

**Ответ:**
```json
{
  "success": true,
  "entries": [
    {
      "id": "kn_01JABCDEF...",
      "kind": "finding",
      "content": "Djinni RSS feed доступен по адресу djinni.co/jobs/feed/ без авторизации. 47 вакансий извлечено.",
      "tags": ["djinni", "rss", "scraping"],
      "confidence": "high",
      "agentName": "Scout",
      "taskTitle": "Scrape Djinni vacancies",
      "createdAt": 1700000000000,
      "daysAgo": 12
    }
  ],
  "count": 1,
  "totalInBase": 143
}
```

Агент добавляет этот вызов в начало своей работы по необходимости:
```bash
curl -s "http://localhost:3001/api/internal/query-knowledge?teamId=${TEAM_ID}&tags=scraping&limit=5"
```

#### 4. Обновление CLAUDE.md: явная документация endpoint'а

В `ClaudeMdGenerator` добавляется секция `### Query team long-term memory` в инструментарий воркера и лидера. Сама база **не инжектируется** в CLAUDE.md автоматически — агент запрашивает её по необходимости, что исключает раздувание контекста.

#### 5. Confidence degradation (фоновая задача)

`KnowledgeMaintenanceService` (cron, раз в сутки):
- Записи старше 30 дней: `high → medium`
- Записи старше 90 дней: `medium → low`
- Записи старше 365 дней с confidence=low: помечаются как `archived` (не удаляются — для UI истории)

#### 6. UI: «Knowledge Timeline» — новая вкладка в TeamDetail

Новая вкладка «Memory» рядом с «Chat» и «Tasks»:
- **Timeline View**: хронологический поток knowledge entries, сгруппированных по сессиям
- **Filter Bar**: фильтр по kind, tags, confidence, дате
- **Entry Card**: показывает kind-иконку, содержание, имя агента-автора, задачу-источник (кликабельно), возраст, confidence badge
- **Session Grouping**: записи из одной сессии схлопываются под заголовком «Session — [дата]»
- **Search**: fuzzy-поиск по содержимому записей
- **Export**: JSON-экспорт всей базы знаний команды

**Визуальный дизайн**: dark theme, цветовая кодировка по kind (finding=синий, decision=фиолетовый, artifact=зелёный, failure=красный), confidence badge (серый для low, стандартный для medium, подсвеченный для high).

### Сценарий использования (UX/Flow)

#### Акт I: Первый запуск (Job Hunting команда, неделя 1)

Пользователь создаёт команду «Job Hunt Alpha» с агентами Scout (scraper), Analyst (data) и Luna (leader). Задаёт задачу: «Найди 50 вакансий Python-разработчика на Djinni и DOU».

1. Scout исследует источники. Выясняет: Djinni работает через RSS (без авторизации), DOU требует FlareSolverr на порту 8191.
2. Scout вызывает `update-task` с summary: «Djinni RSS: djinni.co/jobs/feed/ — 47 вакансий. DOU требует FlareSolverr (localhost:8191). Сохранено в project/vacancies.json».
3. **Автоматически**: `KnowledgeExtractorService` создаёт 2 записи:
   - `kind=finding`, `content="Djinni RSS feed работает без авторизации"`, `tags=["djinni","rss","scraping"]`
   - `kind=finding`, `content="DOU блокирует парсинг, требуется FlareSolverr"`, `tags=["dou","scraping","flaresolver"]`
4. В UI появляется вкладка «Memory» с первыми двумя записями.

#### Акт II: Второй запуск (неделя 2)

Пользователь возвращается: «Найди новые вакансии за эту неделю».

1. Luna назначает задачу Scout.
2. Scout (в начале работы, руководствуясь CLAUDE.md): `curl /api/internal/query-knowledge?teamId=X&tags=scraping&limit=5`
3. Получает: записи с Djinni RSS и FlareSolverr. **Scout пропускает фазу разведки** — сразу идёт к известным источникам.
4. Экономия: 1–2 Claude API-вызова, 3–5 минут реального времени.
5. Analyst также запрашивает базу знаний: находит, что прошлая сессия уже отфильтровала компании X, Y, Z.

#### Акт III: Неожиданный провал (неделя 3)

DOU изменил структуру, FlareSolverr больше не помогает. Scout получает ошибку, вызывает `update-task` со статусом `failed`. Запись автоматически создаётся с `kind=failure`, `confidence=high`, `tags=["dou","scraping"]`.

В следующей сессии Luna и Scout видят в базе знаний:
- `[HIGH] failure (2d ago): DOU scraping через FlareSolverr провалился — 403 Forbidden`

Команда адаптируется, не повторяя провал вслепую.

#### Акт IV: Пользователь как историк (любой момент)

Пользователь открывает вкладку «Memory»:
- Видит 34 записи за 3 недели
- Фильтрует по `kind=failure` — видит 3 провала, их контекст и временну́ю точку
- Фильтрует по `tags=djinni` — видит эволюцию знаний об источнике
- Кликает на запись → попадает в задачу-источник → видит полный trace агента
- Нажимает «Export JSON» → получает полную базу знаний команды для анализа

### Конкурентное позиционирование

На момент написания (апрель 2026) ни один публично известный локальный multi-agent runtime не реализует **персистентную автоматическую базу коллективных знаний** команды агентов, накапливаемую между сессиями без ручной инжекции. Gamma Runtime с GTLM — первый.

Ближайшие аналоги:
- **LangGraph**: хранит состояние *внутри* одного пайплайна, но не между запусками
- **CrewAI**: memory-модуль существует, но требует ручного подключения и не обеспечивает UI-визуализацию
- **Letta (MemGPT)**: специализируется на памяти одного агента, не команды

GTLM даёт Gamma Runtime уникальное позиционирование: **единственная платформа, где команда AI-агентов обучается со временем** — без fine-tuning, без векторных баз данных, без сложной инфраструктуры.
