## I. Гравитационное поле проекта

Прежде чем войти в итерационный цикл, необходимо зафиксировать ограничения, формирующие пространство поиска. Эти инварианты были вычленены из исходного кода и выступают в роли «гравитационной константы» — любая концепция, противоречащая им, отвергается без обсуждения.

1. Агент — это не HTTP-вызов к удалённой модели, а полноценный OS-процесс, порождённый `child_process.spawn('claude', ...)` с `--output-format stream-json`.
2. Координация агентов осуществляется исключительно через локальный REST API (`/api/internal/*`), SSE-канал и разделяемую файловую систему `data/workspaces/{teamId}/`.
3. Шина событий — in-memory `EventEmitter2` (никакого Redis, никакого брокера). Единственный инстанс.
4. Субстрат идентичности агента — `CLAUDE.md`, генерируемый динамически перед каждым запуском.
5. Постгрес хранит иммутабельный `trace`-лог всех событий.
6. Платформа — macOS; CRIU, BPF-трассировка и аналогичные Linux-only механизмы недоступны.
7. V8/Node.js v22 — основной рантайм. Локальные модели допустимы через `node-llama-cpp` (GGUF), но не как жёсткая зависимость.
8. Нет внешнего LLM-шлюза: идентичность модели агентов совпадает с идентичностью модели пользователя (Claude).

## III. Уцелевший: Апофатический Агент (Via Negativa Coordination)

### 3.1 Центральное наблюдение

Все известные мульти-агентные архитектуры для LLM — в том числе две концепции, разработанные в предыдущих итерациях настоящего протокола, — организуются вокруг *позитивного* первичного сигнала. Агент декларирует: «Я могу это», «Я сделал это», «Это работает». Маршрутизация, память, консенсус, отладка — всё строится на положительных утверждениях с последующей верификацией.

При этом ни в одной известной системе *отрицание* не выступает в роли первоклассной, композируемой, разделяемой единицы координации. Отрицание встречается как побочный продукт (рефлексия неудач, отчёты об ошибках, негативные примеры в тренировочном наборе), но не как первичный координационный артефакт.

Апофатическая оркестрация переворачивает это соотношение. Она требует, чтобы каждый агент после каждого хода производил **Манифест Отрицаний** — формализованный, композируемый, разделяемый стек высказываний вида «решение задачи T не лежит в подпространстве S, по причине E». Эти манифесты накапливаются, образуя сжимающийся образ пространства решений. Задача считается разрешённой не тогда, когда один из агентов *декларировал* ответ, а когда пересечение всех накопленных отрицаний оставляет единственную возможность — и ответ всплывает как **остаточная неотрицаемая область**.

### 3.2 Формальное определение

Пусть `T` — открытая задача, а `Ω_T` — гипотетическое (неявное, неизмеримое напрямую) пространство её возможных решений. Для каждой задачи определим:

```
N(T, t) = { n₁, n₂, ..., nₖ } — стек отрицаний, накопленный к моменту t
```

Каждое отрицание n_i имеет формальную структуру:

```
n_i = (classifier, statement, evidence, confidence, discovered_by, scope, epistemic_type)
```

где:

| Поле | Семантика |
|---|---|
| `classifier` | `solution_space` | `method` | `assumption` | `data_source` | `constraint` |
| `statement` | Естественно-языковое высказывание формы «X не является/не верно/не применимо, поскольку Y» |
| `evidence` | Ссылка на событие `trace_log`, файл с результатом теста, вывод инструмента |
| `confidence` | Значение в [0, 1]; высокая уверенность присваивается лишь при эмпирической проверке |
| `discovered_by` | ULID агента-автора |
| `scope` | `this_task` | `this_team` | `global` (область релевантности отрицания) |
| `epistemic_type` | `empirical` (проверено инструментом), `logical` (выведено из инварианта), `heuristic` (экспертная интуиция) |

Определим операцию **пересечения отрицаний**:

```
S_T = Ω_T \ ⋃_{i=1..k} region(n_i)
```

где `region(n_i)` — подпространство `Ω_T`, исключаемое отрицанием n_i. Задача считается **сходящейся**, если `|S_T| = 1` (существует единственная неотвергнутая гипотеза), или **противоречивой**, если `S_T = ∅` (все гипотезы отвергнуты — задача либо неразрешима, либо в стеке есть противоречащие отрицания).

### 3.3 Семь архитектурных инвариантов

**Инвариант 1. Двойственность вывода.** Каждый ход агента обязан завершаться *двумя* артефактами: `positive_emission.md` (традиционный результат) и `via_negativa.jsonl` (новая обязательная единица). Второй артефакт может быть пустым, но отсутствие его поля в трейсе считается ошибкой протокола.

**Инвариант 2. Аддитивная алгебра.** Отрицания композируются через логическое И: `N_1 ∧ N_2 ∧ ... ∧ N_k`. Это фундаментально отличается от композиции позитивных утверждений, требующей семантического слияния, разрешения конфликтов и consensus-протоколов. Два агента, независимо работающие над одной задачей, могут безопасно сливать свои отрицания без семантического арбитража.

**Инвариант 3. Маршрутизация по градиенту неисследованного.** Лидер-агент (или автоматический оркестратор) выбирает подзадачу не по критерию «какой агент лучше справится», а по критерию «в какой области пространства решений ещё нет отрицаний». Следующая подзадача порождается как зонд в максимально «тёмной» области.

**Инвариант 4. Метрика прогресса — коэффициент сжатия (shrinkage ratio).** Формально:

```
shrinkage(T, t) = 1 − |S_T(t)| / |Ω_T|
```

Поскольку `|Ω_T|` неизмерима напрямую, её оценивает лёгкая локальная модель (см. раздел V), сравнивая размер пространства гипотез «до» и «после» добавления каждого отрицания. Метрика выставляется в SSE и визуализируется в UI.

**Инвариант 5. Провал как вклад.** Агент, не выполнивший задачу, всё равно производит отрицания формата «я попробовал подход X, он не сработал по причине Y». В классической мульти-агентной архитектуре такой агент представляет собой чистые потери (затраченные токены без результата). В апофатической архитектуре его вклад положителен: он исключил подпространство X из `Ω_T`. Это радикально меняет экономику неудач в мульти-агентных системах.

**Инвариант 6. Иммутабельность стека.** Отрицания не удаляются и не перезаписываются. Если агент позже обнаружит, что предыдущее отрицание было ошибочным, он публикует *контр-отрицание* — отрицание отрицания (`n_j` = «отрицание n_i неверно, поскольку Z»), сохраняя полную историю эпистемической эволюции команды. Это сближает архитектуру с блокчейн-логикой иммутабельности, но без распределённого консенсуса.

**Инвариант 7. Эмержентность ответа.** Задача не «завершается» в традиционном смысле. Когда `shrinkage → 1`, лидер-агент генерирует *описание остаточной области* — то, что не было отвергнуто, — и предъявляет это как решение. Решение, таким образом, никогда не утверждается напрямую; оно возникает как «единственное, что уцелело».

### 3.4 Иллюстративный сценарий

Пусть пользователь ставит задачу: «Почему запрос к таблице `events` замедлился после вчерашнего деплоя?».

Классическая мульти-агентная команда начнёт выдвигать гипотезы: «Это индекс», «Это параметры сессии», «Это новый код в хот-пути». Проверка гипотез последовательна; если первая гипотеза неверна, время потеряно.

Апофатическая команда порождает параллельные зонды, каждый из которых не *утверждает* причину, а *исключает* её:

- **Агент A** (via negativa): «Замедление НЕ вызвано деградацией дискового I/O: `iostat` показывает стабильную утилизацию за сутки. Уверенность 0.9, эмпирическое.»
- **Агент B** (via negativa): «Замедление НЕ связано с ростом объёма таблицы: count(*) не изменился за 24ч. Уверенность 0.95, эмпирическое.»
- **Агент C** (via negativa): «Замедление НЕ вызвано изменением планов выполнения: `EXPLAIN ANALYZE` возвращает тот же план, что и в предыдущей версии. Уверенность 0.85, эмпирическое.»
- **Агент D** (via negativa): «Замедление НЕ вызвано сетевой latency: измерения ping к db-хосту стабильны. Уверенность 0.8, эмпирическое.»
- **Агент E** (via negativa): «Замедление НЕ обусловлено изменением параметров Postgres: diff `pg_settings` пуст. Уверенность 0.99, логическое.»
- **Агент F** (via negativa): «Замедление НЕ связано с конкурирующими транзакциями: `pg_stat_activity` не показывает блокировок. Уверенность 0.9, эмпирическое.»

После шести отрицаний пространство `Ω_T` сжимается. Лидер-агент запрашивает локальную модель: «Что остаётся?». Ответ: «Остаётся изменение в приложении, порождающее дополнительный roundtrip или дополнительный RLS-предикат на уровне клиента». Эта область становится предметом финального, уже целенаправленного зонда.

Экономия состоит в том, что *отрицания были распараллелены*, а поиск решения свёлся к остатку. Классический подход потратил бы те же инструменты последовательно, в погоне за «правильной» гипотезой.

## IV. Научная Новизна

Представляется необходимым показать, что концепция Апофатического Агента не сводится ни к одному из известных направлений. Ниже приводится последовательная декомпозиция потенциально близких подходов.

### 4.1 Отличие от Reflexion (Shinn et al., 2023)

Reflexion реализует самокритику агента на основе истории его собственных неудач. Рефлексия локальна (в рамках одной сессии одного агента) и используется как сигнал для самокоррекции. Апофатическая архитектура экстернализует отрицания как **первоклассную разделяемую БД** с композиционной алгеброй; отрицания становятся средством *межагентной* координации, а не лишь *интраагентной* самокорректировки. Изменяется единица коммуникации: не «я неверно думал», а «пространство решения не содержит X».

### 4.2 Отличие от Multi-Agent Debate (Du et al., 2023)

Дебаты между агентами — состязательный протокол: агенты производят взаимоисключающие позитивные утверждения, вынуждая друг друга уточняться. Апофатическая архитектура **не состязательна, а аддитивна**: агенты не оппонируют, а совместно сокращают пространство. Сужение монотонно: каждое новое отрицание только увеличивает shrinkage, никогда не уменьшает его (за исключением явных контр-отрицаний, которые трактуются отдельно).

### 4.3 Отличие от Uncertainty Quantification в LLM (Kadavath et al., 2022, arXiv:2207.05221)

Работа «Language Models (Mostly) Know What They Know» исследует калибровку уверенности модели в *позитивных* ответах — насколько хорошо вероятность модели предсказывает корректность её утверждений. Апофатическая архитектура оперирует уверенностью в *отрицаниях*, обладающих фундаментально иным эпистемическим свойством: отрицания монотонно композируются, тогда как позитивные утверждения при композиции порождают конфликты.

### 4.4 Отличие от Constraint Logic Programming (Prolog, MiniZinc, Z3)

CLP оперирует заранее заданной системой ограничений и ищет удовлетворяющие её решения. Апофатическая архитектура **генеративно выводит ограничения в runtime** из эмпирического опыта LLM-агентов. Это не поиск решения в фиксированном constraint-пространстве, а построение самого constraint-пространства по мере исполнения.

### 4.5 Отличие от SAT-solver clause learning (CDCL, Marques-Silva & Sakallah, 1999)

Ближайшая техническая аналогия. CDCL обучается конфликтным клаузам при backtrack, накапливая ограничения, исключающие уже пройденные тупики. Критическое отличие: CDCL оперирует над **синтаксически фиксированной булевой формулой**; пространство гипотез предопределено и дискретно. Апофатическая архитектура оперирует над **открытым семантическим пространством** задачи реального мира, а отрицания генерируются LLM-агентами из исполнения в реальности (результаты тестов, ответы инструментов, проверки состояния).

### 4.6 Отличие от Chain-of-Verification (Dhuliawala et al., 2023, arXiv:2309.11495)

CoVe — интраагентный протокол: модель сама себя проверяет, задавая верификационные подвопросы. Проверка и ответ выполняются одним агентом на одной LLM. Апофатическая архитектура — межагентный, иммутабельный, разделяемый стек; её единица — не «проверка собственного ответа», а «публичная декларация, что область исключена».

### 4.7 Отличие от локальных прецедентов

**Относительно TTT (Temporal Topology of Thinking).** TTT анализирует *форму* когнитивного потока — длины чанков, межприходные интервалы, ритмичность. Апофатическая архитектура работает в пространстве *семантического содержания* — с декларативными высказываниями о том, что исключено. TTT и Via Negativa ортогональны: можно реализовать обе одновременно; TTT скажет «как агент думает», Via Negativa — «что агент исключил».

**Относительно Resonance OS.** Resonance OS использует эмбеддинговое *притяжение* — позитивную семантическую тягу между задачами и агентами. Апофатическая архитектура оперирует *отталкиванием* — накоплением регионов, куда решение *не* попадёт. Противоположные знаки в пространстве координации; концепции взаимодополняемы, а не взаимозаменяемы.

### 4.8 Глубокое сканирование параллелей в биологии, физике и философии

Сильные концепции часто имеют скрытые изоморфизмы в других научных дисциплинах. Настоящий подраздел фиксирует их, чтобы удостовериться, что концепция не является переупаковкой чего-то уже широко известного.

**Теология.** Apophatic theology (via negativa) — интеллектуальная традиция, восходящая к Псевдо-Дионисию Ареопагиту (V в.), по которой абсолют описывается исключительно через отрицания («Бог не есть X, не есть Y...»). Это философская мета-рамка Апофатического Агента, но не техническая реализация. Перенос в область мульти-агентной LLM-оркестрации не обнаружен.

**Эпистемология науки.** Критерий фальсифицируемости Поппера (Popper, «Logik der Forschung», 1934) утверждает, что научная теория характеризуется тем, *что она запрещает*, а не тем, что предсказывает. Апофатический Агент — инженерная проекция фальсификационизма на мульти-агентную систему. Параллель принципиальная, но не реализационная.

**Биология.** Иммунная система позвоночных обучается через *негативную селекцию*: T-клетки, реагирующие на «своё», элиминируются в тимусе. Иммунитет определяет, *что не есть патоген*, а не перечисляет патогены. Биологический изоморфизм есть, но он абстрактен.

**Физика.** Принцип исключения Паули (1925) — фундаментальный закон квантовой механики, конструирующий всю химию через запрет одновременного нахождения двух фермионов в одном состоянии. Вселенная сама по себе апофатична в своём базисе. Параллель метафизична, но не имплементационна.

**Логика.** Силлогизм Шерлока Холмса («Когда вы исключили невозможное, то, что остаётся, сколь бы невероятным оно ни было, — истина» — «Знак четырёх», 1890) — литературное выражение апофатического эпистемического метода. Параллель литературна, но не научна.

Ни одна из перечисленных параллелей не доходит до уровня вычислительной инстанциации в LLM-мультиагентике. Следовательно, инженерное воплощение апофатики как первичного координационного протокола — net-new конструктивный акт.

### 5.1 Расширение схемы БД

Создаётся новая миграция `apps/core/src/database/migrations/002-apophatic.sql`:

```sql
CREATE TABLE negations (
  id               TEXT PRIMARY KEY,           -- neg_01H... (ULID)
  task_id          TEXT NOT NULL REFERENCES tasks(id) ON DELETE CASCADE,
  team_id          TEXT NOT NULL REFERENCES teams(id) ON DELETE CASCADE,
  agent_id         TEXT NOT NULL,
  classifier       TEXT NOT NULL
                     CHECK (classifier IN
                     ('solution_space','method','assumption','data_source','constraint')),
  statement        TEXT NOT NULL,
  evidence_ref     TEXT,
  confidence       REAL NOT NULL CHECK (confidence BETWEEN 0 AND 1),
  scope            TEXT NOT NULL
                     CHECK (scope IN ('this_task','this_team','global')),
  epistemic_type   TEXT NOT NULL
                     CHECK (epistemic_type IN ('empirical','logical','heuristic')),
  parent_negation  TEXT REFERENCES negations(id),   -- для контр-отрицаний
  created_at       BIGINT NOT NULL,
  UNIQUE (task_id, statement)                        -- дедупликация внутри задачи
);

CREATE INDEX idx_negations_task        ON negations(task_id);
CREATE INDEX idx_negations_team_scope  ON negations(team_id, scope);
CREATE INDEX idx_negations_created_at  ON negations(created_at DESC);

CREATE TABLE task_shrinkage (
  task_id       TEXT PRIMARY KEY REFERENCES tasks(id) ON DELETE CASCADE,
  shrinkage     REAL NOT NULL DEFAULT 0 CHECK (shrinkage BETWEEN 0 AND 1),
  residual_desc TEXT,                           -- описание остаточной области
  updated_at    BIGINT NOT NULL
);
```

### 5.6 Маршрутизатор по градиенту неисследованного

В `OrchestratorService` добавляется метод `selectNextSubtaskApophatic`:

```typescript
async selectNextSubtaskApophatic(taskId: string): Promise<SubtaskProposal> {
  const negations = await this.negationsRepo.listByTask(taskId);
  const task       = await this.tasksRepo.getById(taskId);

  // Формируем промпт: «Вот задача T. Вот области, уже исключённые.
  // Предложи зонд в максимально тёмную (непокрытую) область».
  const proposal = await this.shrinkageService.proposeDarkestProbe(
    task.description,
    negations,
  );

  return {
    title: proposal.title,
    description: proposal.description,
    probe_vector: proposal.vector_hash,
    expected_elimination: proposal.expected_shrinkage_delta,
  };
}
```

Этот метод конкурирует с классическим «leader-driven decomposition» и может быть включён либо как альтернативный режим (team flag), либо как стадия внутри существующего пайплайна (сперва лидер декомпозирует, затем apophatic-маршрутизатор уточняет порядок и цели подзадач).

### 5.10 План валидационного эксперимента

**Гипотеза H1.** Команды с включённой апофатической оркестрацией решают задачи из SWE-Bench-Lite (50 задач) за меньшее количество токенов, чем контрольная группа с классическим leader/worker протоколом.

**Гипотеза H2.** Апофатические команды демонстрируют более высокий success rate на задачах с большим пространством решений (debugging, root-cause analysis), но не обязательно на задачах с маленьким пространством (чисто имплементационных).

**Гипотеза H3.** Средняя доля «полезных» отрицаний (те, что входят в траекторию к финальному решению) превышает случайный baseline — что является индикатором того, что LLM-агенты способны генерировать содержательные фальсифицируемые высказывания, а не тавтологии.

**Измеряемые метрики:**
- Total tokens до завершения задачи
- Wall-clock latency
- Success rate (human-graded)
- Shrinkage monotonicity (доля задач, где shrinkage монотонно растёт без контр-отрицаний)
- Negation-per-minute rate по агентам
- Contradiction frequency (доля задач, в которых обнаружены противоречивые отрицания)

### 5.11 Открытые риски и неразрешённые вопросы

Честное признание границ концепции:

1. **Риск пустых отрицаний.** LLM-агент может скатиться к генерации тавтологий («решение не в Марсе»). Необходимо ввести автоматический фильтр тривиальности, возможно — через оценку «неочевидности» локальной моделью.
2. **Риск infinite-loop отрицаний.** Агент-апофатик может производить бесконечно много низкоценных отрицаний, не приближая задачу к решению. Необходим shrinkage-gate: если очередное отрицание даёт `Δshrinkage < ε`, оно считается шумом и не публикуется.
3. **Противоречия в стеке.** Два агента могут сформулировать логически противоречащие отрицания. Процедура разрешения через `counter-negation` должна быть формализована, а не оставлена на усмотрение лидера.
4. **Оценка `|Ω_T|`.** Пространство решений неизмеримо напрямую. Оценка через лёгкую локальную модель носит эвристический характер; вопрос калибровки этой эвристики остаётся открытым.
5. **Масштабируемость.** При большом количестве задач и агентов стек отрицаний может разрастись. Стратегия пропагации scope=global отрицаний требует отдельной проработки (возможно, через эмбеддинговый индекс для поиска релевантных глобальных отрицаний по новой задаче).

Эти вопросы образуют фронтир последующего исследования и не являются возражениями к самой концепции.

## VI. Заключение

Протокол Нулевого Предшествующего Уровня Техники был прогнан во второй раз, с учётом того, что первая итерация уже породила две содержательные концепции (TTT и Resonance OS), которые следовало включить в кладбище наравне с внешней литературой. Это ужесточило критерий новизны.

После двенадцати циклов отклонения Agent B не смог сопоставить концепцию **Апофатического Агента** ни с одним известным источником: ни с академическими публикациями (Reflexion, Multi-Agent Debate, Uncertainty Quantification, CDCL, Chain-of-Verification), ни с инженерными фреймворками (AutoGen, CrewAI, MetaGPT, LangGraph, DSPy), ни с предыдущими документами настоящего репозитория. Концепция имеет глубокие принципиальные параллели в философии (Поппер, Псевдо-Дионисий), биологии (негативная селекция Т-клеток), физике (принцип Паули), но эти параллели существуют на уровне принципа, а не инстанциации в мульти-агентной LLM-оркестрации.

Инверсия первичного сигнала координации с «позитивного утверждения» на «композируемое отрицание» меняет фундаментальные свойства архитектуры: отрицания аддитивно композируются, провал становится положительным вкладом, прогресс измеряется сжатием пространства решений, а ответ возникает как остаточный, а не декларированный. Ни одно из этих свойств не достижимо в существующих мульти-агентных системах без значительных адаптаций.

Концепция полностью совместима с гравитацией проекта Gamma Runtime v2 и может быть реализована в рамках Node.js v22, V8, локальной GGUF-модели (Qwen 2.5 0.5B через `node-llama-cpp`), существующего Postgres и EventEmitter2. Никаких внешних сервисов, никаких Linux-специфичных зависимостей.

Настоящий документ предлагается к рассмотрению как ортогональное дополнение к TTT (форма мышления) и Resonance OS (семантическое притяжение), образующее, в совокупности, триаду базисных координационных примитивов: **структура — притяжение — исключение**.
